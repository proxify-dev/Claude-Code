## Task Management for Multi-Session Work

<Tip>**Official reference** — For `TaskCreate`, `TaskUpdate`, `TaskList`, and task persistence mechanics, see the [official task management docs](https://docs.anthropic.com/en/docs/claude-code/sub-agents#task-management). This section covers the design patterns.</Tip>

Native task management gives agents persistent, structured work items that survive context compaction and session boundaries. The design decisions:

**Task granularity:** 5-6 tasks per agent is the sweet spot. Fewer means the agent disappears for long stretches with no visibility. More creates overhead that outweighs the coordination benefit.

**Dependencies:** Tasks can block other tasks. Use this to enforce sequencing — the validation task can't start until the implementation task completes. This is how [pipeline chains](../patterns/orchestration-patterns#pipeline-chains) work in practice.

**Multi-session coordination:** Set `CLAUDE_CODE_TASK_LIST_ID` to share a task list across sessions. One agent creates the tasks; others pick them up. This enables the [fan-out](../patterns/orchestration-patterns#fan-out--fan-in) pattern across separate terminal sessions.

**Progress visibility:** Agents update task status as they work. The lead session (or you) can check `TaskList` to see what's done, what's in progress, and what's blocked — without interrupting the working agents.

## Rules Directory Architecture

<Tip>**Official reference** — For `.claude/rules/` file format, path patterns, and discovery mechanics, see the [official rules docs](https://code.claude.com/docs/en/memory#rules-files). This section covers when to use rules vs. CLAUDE.md.</Tip>

CLAUDE.md works well up to ~200 lines. Beyond that, priority saturation sets in — when everything is high-priority, nothing is. The rules directory solves this with path-scoped context.

| Problem | CLAUDE.md | `.claude/rules/` |
|---------|-----------|-------------------|
| Global conventions (imports, naming) | Put it here | Overkill |
| Area-specific rules (API testing, frontend patterns) | Gets cluttered | Scoped by path pattern |
| Team-shared standards | Works if short | Better at scale |

**Path-scoped rules** load only when Claude is working on files that match the pattern:

```
.claude/rules/
  api-testing.md          # globs: src/api/**
  frontend-patterns.md    # globs: src/components/**
  migration-safety.md     # globs: migrations/**
```

Each file loads only when relevant — an agent working on frontend components never sees migration rules, keeping context lean and instructions focused.

**When to split:** If your CLAUDE.md has sections that only apply to specific areas of the codebase, move them to rules files. Keep CLAUDE.md for truly global conventions.

## Agent Context Strategy

Agents don't share your context. Every agent starts fresh with only what you provide in the invocation plus CLAUDE.md and rules. Design for this:

- **Self-contained invocations** — include file paths, success conditions, and boundaries. See [invocation quality](../patterns/invocation-quality/).
- **CLAUDE.md as shared context** — conventions, module boundaries, and verification commands that every agent needs
- **Skills for complex workflows** — if an agent needs a multi-step process (deploy, migrate, review), encode it as a [skill](../skills/creating-skills/) rather than inlining it in every invocation
- **Progress files for long-running work** — a `PROGRESS.md` that agents update gives fresh agents (post-compaction or new session) a way to resume

The goal: any agent should be able to start working with just the invocation + CLAUDE.md + relevant rules, without needing your conversation history.
