## File Location

Place agent files in the `agents/` directory of your plugin:

```
my-plugin/
└── agents/
    ├── code-reviewer.md
    ├── test-generator.md
    └── docs-writer.md
```

All `.md` files in `agents/` are auto-discovered.

## Frontmatter Fields

```yaml
---
name: code-reviewer          # Required. Lowercase, hyphens, 3-50 chars.
description: Use this agent when...  # Required. Triggering conditions + examples.
model: inherit               # Required. inherit | sonnet | opus | haiku
color: blue                  # Required. blue | cyan | green | yellow | magenta | red
tools: ["Read", "Grep"]      # Optional. Restricts available tools.
disallowedTools: ["Write"]   # Optional. Denies specific tools.
mcpServers:                  # Optional. MCP server access.
  - github
permissionMode: default      # Optional. default | acceptEdits | dontAsk | bypassPermissions | plan
maxTurns: 20                 # Optional. Limits agent turns.
skills: ["my-skill"]         # Optional. Skills available to the agent.
memory: project              # Optional. Persistent memory scope: user | project | local
effort: medium               # Optional. Reasoning depth: low | medium | high | max
background: false            # Optional. Run as background agent by default.
isolation: worktree          # Optional. Git-isolated working directory.
---
```

<Tip>Use the `/agents` command for an interactive interface to create and manage agent definitions — it's easier than editing files manually.</Tip>

### `name`

Lowercase letters, numbers, and hyphens only. Must start and end with alphanumeric.

### `description`

The most critical field — determines when Claude triggers this agent. Must include triggering conditions and `<example>` blocks:

```
Use this agent when [conditions]. Examples:

<example>
Context: [Scenario]
user: "[What user says]"
assistant: "[How Claude responds]"
<commentary>
[Why this agent is appropriate]
</commentary>
</example>
```

Include 2–4 examples covering different phrasings and proactive/reactive triggering.

### `model`

| Value | Use when |
|-------|----------|
| `inherit` | Default — uses the parent's model |
| `sonnet` | Balanced speed and capability |
| `opus` | Complex analysis, highest capability |
| `haiku` | Fast and cheap, simple tasks |

### `color`

Visual identifier in the UI. Choose colors semantically:

| Color | Agent type |
|-------|-----------|
| Blue / Cyan | Analysis, review |
| Green | Generation, creation |
| Yellow | Validation, caution |
| Red | Security, critical |
| Magenta | Creative, transformation |

### `tools`

Restricts which tools the agent can use. Omit for full access.

| Use case | Tool set |
|----------|----------|
| Read-only | `["Read", "Grep", "Glob"]` |
| Code generation | `["Read", "Write", "Grep"]` |
| Testing | `["Read", "Bash", "Grep"]` |

### `disallowedTools`

Explicitly denies specific tools. Useful for [builder-validator patterns](../patterns/orchestration-patterns#builder-validator-pairs) where you want near-full access minus writes:

```yaml
disallowedTools: ["Write", "Edit", "NotebookEdit"]
```

### `permissionMode`

| Value | Behavior |
|-------|----------|
| `default` | Normal permission prompting |
| `acceptEdits` | Auto-approves file edits, prompts for other operations |
| `dontAsk` | Skips most permission prompts |
| `bypassPermissions` | No permission checks |
| `plan` | Read-only, no modifications |

### `background`

Set to `true` to run the agent in the background by default. You'll be notified when it completes. Useful for long-running tasks like test suites or large analysis jobs.

### `isolation`

Set to `"worktree"` to run the agent in a temporary git worktree. Each agent gets its own branch and working directory — essential when multiple agents might edit the same files. See [worktree isolation](../patterns/parallel-execution#worktree-isolation-on-agents).

### `effort`

Controls reasoning depth: `low`, `medium`, `high`, or `max`. Lower effort means faster, cheaper responses. Higher effort means more thorough analysis.

### `mcpServers`

**Inline** — scoped to the subagent; spins up fresh, main context never sees it:
```yaml
mcpServers:
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@playwright/mcp@latest"]
```

**String reference** — reuses the parent session's existing connection:
```yaml
mcpServers:
  - github
```

## System Prompt

The markdown body after the frontmatter becomes the agent's system prompt. Write in second person.

### Template

```markdown
You are [role] specializing in [domain].

**Your Core Responsibilities:**
1. [Primary responsibility]
2. [Secondary responsibility]

**Process:**
1. [Step one]
2. [Step two]
3. [Step three]

**Output Format:**
- [What to include]
- [How to structure]

**Edge Cases:**
- [Edge case 1]: [How to handle]
```

### Tips

- Be specific about responsibilities — avoid generic instructions
- Define a clear step-by-step process
- Specify the output format so results are consistent
- Address edge cases the agent might encounter
- Keep under 10,000 characters
