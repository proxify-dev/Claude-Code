## Writing Descriptions That Trigger

The `description` field determines when Claude routes work to your agent. Generic descriptions fail silently — Claude just never picks the agent.

**Include `<example>` blocks** with 2-4 scenarios covering different phrasings and both proactive and reactive triggers:

```
Use this agent when [conditions]. Examples:

<example>
Context: [Scenario]
user: "[What user says]"
assistant: "[How Claude responds]"
<commentary>
[Why this agent is appropriate]
</commentary>
</example>
```

<Note>**Auto-routing remains unreliable.** Even good descriptions get missed. The most reliable trigger is explicit instruction: "Use the security-reviewer agent to audit this module." Design descriptions for the best case, but tell users to invoke by name when it matters.</Note>

## Color Semantics

Choose colors that signal intent at a glance:

| Color | Agent type |
|-------|-----------|
| Blue / Cyan | Analysis, review |
| Green | Generation, creation |
| Yellow | Validation, caution |
| Red | Security, critical |
| Magenta | Creative, transformation |

## System Prompt Template

The markdown body after frontmatter becomes the system prompt. A structure that works:

```markdown
You are [role] specializing in [domain].

**Core Responsibilities:**
1. [Primary responsibility]
2. [Secondary responsibility]

**Process:**
1. [Step one]
2. [Step two]
3. [Step three]

**Output Format:**
- [What to include]
- [How to structure]

**Edge Cases:**
- [Edge case 1]: [How to handle]
```

Tips:
- Be specific about responsibilities — vague prompts produce vague agents
- Define output format so results are consistent across invocations
- Keep under 10,000 characters — longer prompts dilute attention
